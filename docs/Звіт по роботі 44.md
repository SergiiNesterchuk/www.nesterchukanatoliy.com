# Звіт по роботі #44 — Створення тестового сайту (TestoviySite)

**Дата:** 2026-05-06
**Виконавець:** Claude Opus 4.6 (Claude Code)
**Проєкт:** nesterchukanatoliy.com (Next.js + Prisma + Railway + KeyCRM + WayForPay)

---

## 1. Мета роботи

Створити повністю ізольований тестовий сайт для безпечного тестування нового функціоналу. Тестовий сайт має бути ідентичний production за контентом (товари, категорії, фото, налаштування), але з відключеними реальними платежами (WayForPay) та CRM-синхронізацією (KeyCRM).

**Вимоги:**
- Production сайт не має змінитися для кінцевих користувачів
- Тестовий сайт — окреме середовище з окремою базою даних
- Реальні платежі неможливі на тестовому сайті
- Реальна синхронізація з KeyCRM неможлива на тестовому сайті
- При тестуванні checkout — зрозумілий alert що це тестовий режим
- Можливість синхронізувати контент з production одним кліком

## 5. Ізоляція: як гарантується безпека production

### 5.1. WayForPay (реальні платежі)

**Рівень 1 — env guard у коді:**
`PAYMENTS_ENABLED === "false"` → `OrderService.createPaymentForOrder()` повертає `null` (рядок 149 OrderService.ts)

**Рівень 2 — mock-оплата в checkout:**
Коли payment = null → замовлення автоматично "paid" з `paymentProvider: "test_mode"`, без виклику WayForPay API

**Рівень 3 — невалідні credentials:**
`WAYFORPAY_MERCHANT_ACCOUNT=disabled` → навіть якщо guard обійти, WayForPay API відхилить запит

**Рівень 4 — WayForPayAdapter throw:**
Адаптер кидає Error якщо credentials невалідні (рядок 36-39 WayForPayAdapter.ts)

### 5.2. KeyCRM (синхронізація замовлень)

**Рівень 1 — env guard у коді:**
`CRM_SYNC_ENABLED !== "false"` — перевіряється в 7 місцях (checkout ×3, OrderService callback ×2, processPendingSyncs, retry-sync)

**Рівень 2 — cron guard:**
`KEYCRM_STATUS_SYNC_ENABLED === "false"` → cron endpoint повертає `{skipped: true}`

**Рівень 3 — невалідний API key:**
`KEYCRM_API_KEY=disabled` → KeyCRM API повертає 401

**Рівень 4 — processPendingSyncs guard:**
`!process.env.KEYCRM_API_KEY` → пропускає (рядок 652 KeyCRMService.ts). "disabled" — truthy, але API відхилить.

### 5.3. Analytics / SEO

- Google Analytics і Clarity не завантажуються при `NEXT_PUBLIC_APP_ENV === "staging"`
- `<meta name="robots" content="noindex, nofollow">` → пошуковики не індексують тестовий сайт

### 5.4. Email

Email notifications відправляються через `NotificationService.sendOrderConfirmation()` — вони НЕ мають окремого guard. Але mock-оплата в checkout НЕ тригерить notification flow (mock відбувається ДО того як OrderService callback обробить платіж). Якщо потрібна повна ізоляція email — рекомендується додати `EMAIL_ENABLED=false` guard.

### 5.5. Бази даних

| | Production | TestoviySite |
|---|---|---|
| Host | `postgres-xxx.railway.internal` | `postgres-cqgi.railway.internal` |
| DATABASE_URL | Різний | Різний |
| Дані | Реальні замовлення, клієнти | Тільки контент (товари, налаштування) |

Бази повністю ізольовані — різні PostgreSQL instances з різними паролями та хостами.

## 9. Архітектурні рішення

### 9.1. Чому один Railway project з двома environments (а не два проєкти)?
- Простіше для соло-розробника
- Railway environments повністю ізольовані (різні БД, різні env vars)
- Можливість copy variables між environments
- Один білінг

### 9.2. Чому `NEXT_PUBLIC_APP_ENV` а не `NODE_ENV`?
- `NODE_ENV` контролюється Next.js (завжди `production` при build)
- `NEXT_PUBLIC_*` доступний у client-side коді (для banner)
- Окремий від Next.js lifecycle

### 9.3. Чому mock-оплата а не WayForPay sandbox?
- Простіше налаштування (не потрібні тестові credentials WayForPay)
- Alert чітко показує що це тест
- Можна додати WayForPay sandbox пізніше як альтернативу

### 9.4. Чому один R2 bucket для обох сайтів?
- Фото однакові — немає сенсу дублювати
- R2 — read-only для frontend (URL в БД)
- Якщо staging адмін змінить фото — це вплине на production (ризик мінімальний, бо staging для тестування нового коду, не контенту)

### 9.5. Чому PROD_DATABASE_URL — public URL?
- Staging service не має доступу до production internal мережі
- Public URL дозволяє read-only доступ для синхронізації
- Ризик: egress fees від Railway (мінімальні при рідкій синхронізації)

## 10. Відомі обмеження та ризики

| Ризик | Серйозність | Контроль |
|---|---|---|
| `PROD_DATABASE_URL` в staging env vars | Середня | Використовується тільки для read, admin-only endpoint, перевірка NEXT_PUBLIC_APP_ENV |
| Email notifications при mock-оплаті | Низька | Mock-оплата не тригерить email flow (обходить OrderService callback) |
| Staging адмін може змінити фото в R2 | Низька | R2 bucket шариться, але staging — для тестування коду, не контенту |
| `KEYCRM_API_KEY=disabled` замість порожнього | Низька | Railway CLI не дозволяє порожні значення; "disabled" — невалідний API key |
| Build на Railway після push в main | Очікувано | Обидва environments деплоять з main branch автоматично |
| Turbopack + кирилічний шлях | Не впливає | Баг тільки локально; на Railway шляхи англійські |
