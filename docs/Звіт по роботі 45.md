### 1.2. PAYMENTS_MODE

Новий env variable з 4 режимами:

| Mode | WayForPay | Order status | Use case |
|---|---|---|---|
| `live` | Реальний | Через callback | Production |
| `mock` | Не викликається | Auto-paid через `applyMockPayment()` | Staging default |
| `sandbox` | Тестові credentials | Через реальний callback | Майбутнє |
| `disabled` | Не викликається | Залишається pending | Повна ізоляція |

**Backward-compatible:** `PAYMENTS_ENABLED=false` → `mock` mode.

### 1.3. Mock payment через shared flow (`OrderService.applyMockPayment`)

**Було:** checkout напряму ставив `paymentStatus: "paid"` через `prisma.order.update()`. Це обходило payment lifecycle.

**Стало:** `OrderService.applyMockPayment()` проходить той самий шлях що й реальний WayForPay callback:
1. Створює `PaymentEvent` (provider: "mock", eventType: "success")
2. Оновлює order через `OrderRepository.updatePaymentStatus()` — та ж функція що й при реальному callback
3. Записує `OrderStatusHistory` з prefix `[MOCK]`
4. Тригерить `NotificationService.sendOrderConfirmation()` — для тестування email шаблонів
5. CRM sync НЕ тригериться (blocked by `crmSyncEnabled=false`)

**Сценарії:**
| Checkout flow | Mock behavior |
|---|---|
| 100% card payment | paymentStatus: "paid", provider: "mock" |
| COD передплата 200 грн | paymentStatus: "partial_paid", provider: "mock" |

### 1.4. Ручне тестування оплати в адмінці

На сторінці замовлення в адмінці (тільки staging) додано блок **"Тестова оплата"** з 3 кнопками:

| Кнопка | Action | Результат |
|---|---|---|
| Оплата пройшла | `success` | paid / partial_paid + PaymentEvent + StatusHistory |
| Оплата не пройшла | `failure` | failed / prepayment_failed + PaymentEvent + StatusHistory |
| Повернення коштів | `refund` | refunded + PaymentEvent + StatusHistory |

**API:** `POST /api/admin/test-payments/mock-callback`
**Захист:** `adminGuard` + `isStaging` check (production → 403)
**Файл:** `app/api/admin/test-payments/mock-callback/route.ts`
**UI:** `app/admin/orders/[id]/page.tsx` — кнопки з confirm dialog + alert результату

---

### 1.5. PROD_DATABASE_URL hardened protection

**Sync endpoint** (`app/api/admin/sync-from-production/route.ts`):
- `isStaging` check (з centralized flags, не raw env)
- `DATABASE_URL === PROD_DATABASE_URL` → abort з повідомленням
- Коментар у коді: `prodPrisma is used ONLY for read operations`
- Production → 403

**CLI script** (`scripts/sync-to-staging.ts`):
- `stagingUrl === prodUrl` → throw
- `prodUrl.includes("staging")` → throw

**Аудит:** `prodPrisma` використовується тільки для `findMany()` і `findFirst()`. Жодних `create`, `update`, `delete` через production client.

## 3. Production safety

- `npx tsc --noEmit` → 0 errors
